CAP, ACID & BASE Explained

Distributed systems trade-offs — theory, databases, and real-world examples

The CAP Theorem
A distributed system can guarantee only 2 of 3 properties at once. Partitions always happen — so you choose CP or AP.
Consistency
Every read gets the latest write or an error. All nodes see the same data.
Every ATM shows your real balance — no stale data.
Availability
Every request gets a response (may be stale). The system stays up.
Amazon cart always lets you add items, even during a network split.
Partition Tolerance
System keeps running when messages are lost between nodes.
Packets dropped between data centers — system still runs.
Key insight: In real distributed systems, network partitions always happen. The real choice: during a partition, prioritize Consistency (CP) or Availability (AP)?
ACID vs BASE — Two Philosophies
ACID = strict correctness (CP world). BASE = high availability (AP world). Neither is universally better — they serve different needs.

ACID — Every Property in Depth
Each letter solves a specific failure mode. Together they guarantee correct, predictable state under concurrent transactions.
A
Atomicity
All-or-nothing. Every operation succeeds or the entire transaction rolls back — zero partial writes.
Banking: debit ₹5,000 from A AND credit ₹5,000 to B — both succeed or neither. No half-transfers.
BEGIN/COMMIT/ROLLBACKWAL log
C
Consistency
Every transaction moves DB from one valid state to another. Constraints, cascades, triggers satisfied.
Balance cannot go below zero — any violating CHECK constraint is rejected entirely.
CHECK constraintsFK constraints
I
Isolation
Concurrent transactions run as if serial. Intermediate state invisible until commit. Level is configurable.
Two users booking the last IRCTC seat: only one acquires the row lock — the other is blocked or errors.
MVCCRow-level locks
D
Durability
Once committed, data survives crashes. DB flushes to disk via fsync before acknowledging commit.
NEFT "Success" — a server crash 1 second later cannot undo it. WAL redo log has it on disk.

BASE — Every Property in Depth
BASE trades strong consistency for availability. Nodes fail, networks split — data will eventually converge.
BA
Basically Available
Every request gets a response — not always latest data, but never a total outage.
DynamoDB: during regional outage, other regions still serve reads/writes. Cart may be stale — site never fully down.
Multi-region replicas
S
Soft State
State may change without new input. Replicas propagate asynchronously — nodes can temporarily differ.
Mumbai node shows ₹999; Delhi node (not synced) shows ₹1,099 — both valid at their moment in time.
Async replicationGossip protocol
E
Eventually Consistent
Under no new writes, all replicas converge. Window is typically milliseconds to seconds.
Tweet likes: User A sees 1,203; User B sees 1,201. In 300ms both sync to the same count.

Eventually consistent ≠ wrong forever. Fine for catalogs, feeds, analytics. Never for bank balances or ticket booking.

Dirty read
TX1 reads uncommitted data from TX2. If TX2 rolls back, TX1 read data that never existed.

You see ₹50,000 credited before TX commits — then it rolls back.
Non-repeatable read
TX1 reads a row, TX2 modifies and commits, TX1 re-reads and gets a different value.

Balance report reads ₹10,000 then ₹5,000 in one transaction.
Phantom read
TX1 runs range query, TX2 inserts matching rows, TX1 re-runs and sees phantom rows.

IRCTC queries seats → 3. Another TX books one. Re-query returns 2.
SERIALIZABLE ensures TX2 cannot hold a stale snapshot once TX1 commits. No double booking.

How ACID + BASE Connect to CAP
ACID and BASE are database-level implementations of the CAP trade-off. ACID = CP. BASE = AP. They are consequences of your CAP choice.
CP + ACID = banking
During partition: block writes, return error. No dirty reads, rollback on crash, fsync durability.
PostgreSQLOracleMySQL
AP + BASE = e-commerce / social
During partition: accept writes locally, replicate later. Eventual sync, vector clocks for conflicts.
CassandraDynamoDBCouchDB
NewSQL (CockroachDB, Spanner, YugabyteDB) achieves distributed ACID via Raft/Paxos — CP with very high practical availability.
Banking — CP (Consistency + Partition Tolerance)
Banks sacrifice availability for consistency. A failed transaction beats double-spending ₹50,000. ACID, 2PC, synchronous replication.
CP — synchronous replication, strict ACID
Tech stack

Amazon Cart — AP (Availability + Partition Tolerance)
Amazon chose availability. A cart that's always addable beats one that's offline. Eventual consistency + vector clocks reconcile conflicts.
Tech stack

IRCTC — Hybrid (CP for seats, AP for browsing)
Strict CP for seat allocation (no double booking), AP for timetable browsing. Common in high-traffic booking systems.
